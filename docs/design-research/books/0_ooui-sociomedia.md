## 概要

**OOUI（Object-Oriented User Interface）** とは、タスク（動詞）ではなくオブジェクト（名詞）を起点としてUIを設計する手法。世の中の業務アプリの約8割はタスク指向であり、OOUIに変えるだけで劇的に改善できるとされる。

タスク指向UIに対し、OOUIには以下のメリットがある：

- **画面数が大幅に減少する**（タスク指向で100画面あったものが5〜20画面になる事例もある）
- 操作効率・学習効率・業務効率が向上する
- UIの拡張性・保守性が高まる
- フロントエンドとバックエンドの設計が一致しやすくなる（RESTful APIとの対応）

### OOUIデザインフロー

```
① タスク列挙
   ユーザーが行うタスクを動詞文で書き出す
   例）「スタッフのロールを変更する」「チームに配属する」
        ↓
② オブジェクト抽出（名詞を拾う）
   タスク文から名詞を取り出し、主オブジェクトを特定する
   例）スタッフ / ロール / チーム
        ↓
③ ビュー設計
   各オブジェクトに「コレクションビュー（一覧）」と
   「シングルビュー（詳細）」のペアを設計する
        ↓
④ ナビゲーション設計
   ルートナビゲーション = 主オブジェクト種別（名詞ラベル）
   遷移パターン = コレクション → シングル
        ↓
⑤ アクション配置
   各ビューのコンテキストアクション（動詞）を定義する
   例）コレクション選択時：[削除] [エクスポート]
       シングル表示時：[ロール変更] [チーム変更]
        ↓
⑥ レイアウト作成・検証
   デバイスサイズに合わせてレイアウトを作成し、
   オブジェクトへの到達可能性をユーザー視点で検証する
   （①〜⑥を反復しながら最適なUIに近づける）
```

## 歴史的背景

OOUIの思想は50年以上の歴史を持つ。現代の「オブジェクト指向UI」という考え方に至るまでの系譜は以下の通り。

### 技術的源流の年表

| 年代 | 出来事 |
|------|--------|
| 1963年 | Ivan Sutherlandの**Sketchpad**：ベクターグラフィクスの直接操作を最初に実現 |
| 1968年 | Douglas Engelbartの**NLS**：マウス操作、ハイパーリンク、作業空間としてのスクリーン |
| 1972年 | **Smalltalk**（Xerox PARC）：「すべてはオブジェクトである」という純粋OOP言語、MVCパターンの源流 |
| 1981年 | **Xerox Star**：商業初のGUI搭載パーソナルコンピュータ。デスクトップメタファー、アイコン、直接操作の実現 |
| 1983年 | **Apple Lisa** / 1984年 **Macintosh**：OOUIが商業的に大成功。デスクトップ・フォルダ・ファイルが一般化 |
| 1987-1992年 | **IBM Common User Access（CUA）**：OOUI設計基準として最も包括的な規格。1992年第3版で「Object-Oriented Workplaceモデル」を提唱 |
| 1988年 | **Sun Microsystems OPEN LOOK**：UNIX系OOUIの標準規格 |
| 1990-1993年 | **Microsoft OLE 2.0**：コンパウンドドキュメント、インプレース編集、ドラッグ&ドロップ |
| 1995年 | Alan Cooper『About Face』初版：Goal-Directed DesignとOOUIの思想が融合 |
| 1995年 | Dave Collins『Designing Object-Oriented User Interfaces』（Addison-Wesley）：OOUI初の体系的専門書 |
| 2013年 | Sophia PraterがA List Apart誌で**OOUX（Object-Oriented UX）**を発表 |
| 2020年 | ソシオメディア『オブジェクト指向UIデザイン』：日本語で初の実践書 |

### Xerox Star（1981年）の設計原則

Xerox Starは現代OOUIの直接的な祖先。設計者David Canfield Smithが確立した以下の原則が今日のOOUIに引き継がれている：

- **知覚可能性**：操作対象が画面上に目に見える形で存在する
- **直接操作**：オブジェクトに対してマウスで直接働きかけられる
- **WYSIWYG**：表示と出力が一致している

### IBM CUA 1992年版の革新

IBMのCUA第3版（1992年）は「Object-Oriented Interface Design: IBM Common User Access Guidelines」として出版され、以下の転換を宣言した：

> 「ユーザーインタラクションの主眼を『アプリケーション操作』から『ユーザーが実際に扱うデータ（ドキュメント、画像など）の操作』へ転換する」
> 「アプリケーションを操作してドキュメントを作る」ではなく「ドキュメントを扱うためにアプリケーションを使う」

この思想転換がOS/2の「Object-Oriented Workplaceモデル」に反映された。

## OOUIの4つの原則

1. **知覚可能性（Perceivability）**：操作対象であるオブジェクトが画面上に可視化されており、直接指示できる状態にある
2. **自己表現性（Self-expression）**：各オブジェクトが自身の性質・現在の状態をリアルタイムに示す
3. **操作順序（Interaction Order）**：「名詞→動詞」の構文。先にオブジェクトを選び、次にアクションを選ぶ（タスク指向とは逆）
4. **協調構成（Collaborative Composition）**：すべてのオブジェクトが相互に関連し、論理的な空間を構成する

## 設計プロセスの3ステップ

### Step 1：オブジェクトの抽出

ユーザーが行うタスクを動詞文で列挙し、文中の**名詞を抽出**することで主要オブジェクトを特定する。

```
タスク例：
- スタッフのロールを確認する
- チームへの配属を変更する
- スタッフのスキルを登録する

↓ 名詞を抽出

オブジェクト：スタッフ、ロール、チーム、スキル
主オブジェクト：スタッフ（繰り返し登場する最重要オブジェクト）
従属プロパティ：ロール、チーム、スキル（スタッフの属性）
```

温泉宿アプリの例では「宿」「温泉」「泉質」がオブジェクトとして抽出される。

### Step 2：ビューの設計

各主オブジェクトに対して2種類のビューを設計する：

| ビュータイプ | 説明 | 例 |
|-------------|------|-----|
| **コレクションビュー** | オブジェクトの一覧（複数表示） | スタッフ一覧、メール受信トレイ |
| **シングルオブジェクトビュー** | 1つのオブジェクトの詳細情報 | スタッフ詳細・編集画面 |

ルートナビゲーション（タブバー・サイドバー）のラベルは**名詞形**で表現するのがOOUI的に正しい設計。

### Step 3：レイアウト作成

デバイスサイズに最適化して実際の画面配置を作成し、**オブジェクトへの到達可能性をユーザー視点で検証**する。ステップ1〜3を行き来しながら最適なUIに近づける。

---

## コレクションビューのパターン

コレクションビューは「複数のオブジェクトの集合を俯瞰する」ビュー。Notionのように「同一オブジェクト集合を複数のビューで表示できる」のがモダンOOUIの特徴。

| パターン | 特徴 | 適用例 |
|---------|------|-------|
| **リスト（List）** | 縦に並んだテキスト中心のアイテム行。スキャンと検索に優れる | メール受信トレイ、Slackチャンネル一覧、Linear issue一覧 |
| **グリッド（Grid）** | 等間隔のカードタイル配置。視覚的比較に優れる | iOSフォトアプリ、ショッピングサイト商品一覧、Figmaファイル一覧 |
| **テーブル（Table）** | 複数属性を列として並べる。データ分析・比較に優れる | Notionデータベース、Excelシート、GitHubプルリクエスト一覧 |
| **カンバン（Kanban）** | ステータス列とカードで状態遷移を表現 | Trello、Jira、LinearのBoard view |
| **カレンダー（Calendar）** | 時間軸でオブジェクトを配置 | Google Calendar、Notionカレンダービュー |
| **タイムライン（Timeline）** | 横軸の時系列にオブジェクトを配置 | GitHubロードマップ、Linearタイムライン |

---

## シングルオブジェクトビューのパターン

| パターン | 特徴 | 適用例 |
|---------|------|-------|
| **詳細ビュー（Detail View）** | オブジェクトのすべての属性・関連情報を表示 | GitHub Issue詳細ページ、Linear issue詳細 |
| **フォーム（Form）** | 編集可能なフィールドが並ぶ入力UI | ユーザー設定、プロファイル編集 |
| **インスペクターパネル（Inspector Panel）** | サイドパネルに属性が表示される。選択オブジェクトに応じて内容が動的に変化（コンテキストセンシティブ） | Figma右パネル、Photoshopオプションパネル、Unity Inspector |

**インスペクターパネルの設計ポイント**：選択しているオブジェクトの種類によって表示されるプロパティが動的に変化する「コンテキストセンシティブ」な設計が理想。Figmaではテキストを選択すると文字設定、矩形を選択すると塗りと角丸設定が表示される。

## ナビゲーションパターン

| パターン | 説明 | 適用例 |
|---------|------|-------|
| **マスター詳細（Master-Detail）** | 左列にリスト、右ペインに詳細を並べる分割ビュー | macOSメール、Outlook、Linear |
| **スプリットビュー（Split View）** | 画面を2〜3ペインに分割し階層的に絞り込む | macOS Finder（3カラム表示）、Apple設定アプリ |
| **ドリルダウン（Drill-Down）** | コレクション→サブコレクション→シングルと階層を潜る | iOS設定アプリ、ファイルブラウザ |
| **ルートナビゲーション** | アプリ最上位のナビゲーション。各タブが主要オブジェクト種別に対応 | Slack（チャンネル、DM）、GitHub（Code, Issues, PRs） |

**ルートナビゲーション設計の原則**：タブやサイドバーのラベルは動詞ではなく**名詞形**（オブジェクト種別名）で表現する。

```
❌ タスク指向ナビゲーション例：
   [メールを送る] [ファイルを管理する] [スタッフを登録する]

✅ オブジェクト指向ナビゲーション例：
   [受信トレイ] [ファイル] [スタッフ]
```

## タスク指向 vs オブジェクト指向

| 比較項目 | タスク指向 | オブジェクト指向 |
|---------|-----------|----------------|
| 出発点 | 動詞（やること） | 名詞（もの） |
| UIパターン | ウィザード・フォームの連鎖 | オブジェクト選択→アクション選択 |
| 画面数 | 増えがち（タスク数×ステップ数） | 少なく済む（オブジェクト数×2ビュー） |
| 拡張性 | 低い（新タスク追加でUI増加） | 高い（新プロパティはシングルビューに追加） |
| 操作の自由度 | 線形（ウィザードに縛られる） | 非線形（ユーザーが自由に動ける） |
| 典型例 | ステップ式申請フォーム、ウィザード | macOS Finder、Excel、Notion、GitHub |

## Before/After 変換事例

### 事例1：スタッフ管理システム

**タスク指向（Before）**

```
メニュー：
1. スタッフのロールを変更する
2. チームへの配属を変更する
3. スタッフのスキルを登録する

→ それぞれ別のフォーム画面が存在
→ 同一スタッフへの複数変更に何度もトップに戻る
```

**OOUI（After）**

```
スタッフ一覧（コレクションビュー）
└ 山田太郎 を選択（シングルビュー）
   ├ 基本情報：名前、メール
   ├ ロール：[エンジニア ▼] ← インプレースで変更可能
   ├ チーム：[開発チーム ▼] ← インプレースで変更可能
   └ スキル：[タグ形式で表示・追加可能]
```

### 事例2：メールアプリ

**タスク指向（Before）**

```
トップメニュー：
[新規メールを作成する]
[受信メールを確認する]
[送信済みメールを管理する]
[メールを検索する]
```

**OOUI（After）**

```
受信トレイ（コレクションビュー）
[from: Alice] 来週の会議について  14:30
[from: Bob]   プロジェクト提案書  昨日

→ メールを選択（シングルビュー）
   [返信] [転送] [削除] [アーカイブ]（コンテキストアクション）
```

### 事例3：フードデリバリー（OOUIが適切なケース）

```
✅ OOUI的アプローチ：
ピザ一覧が最初に見える（コレクションビュー）
→ マルゲリータを選択（シングルビュー）
→ 店舗・サイズ・トッピングを選択
→ カートに追加
```

## モダンSaaSにおけるOOUI実装例

### GitHub

- **コレクションビュー**：`/issues`の一覧（リスト・フィルタ可能）
- **シングルビュー**：`/issues/{id}` の詳細ページ
- **インスペクターパネル的UI**：サイドバーでラベル、担当者、マイルストーンを編集
- **ルートナビゲーション**：Code / Issues / Pull Requests / Actions はすべて名詞形

### Linear（プロジェクト管理ツール）

- Issueオブジェクトが中心。同一データをリスト・ボード・タイムラインで切り替え可能
- キーボードショートカット設計もオブジェクトファースト（`G` + `I` で「Issueの一覧へ移動」）
- コンテキストセンシティブなアクションが充実

### Notion

- 「すべてがPageオブジェクト」という設計思想
- データベースは同一オブジェクト集合をテーブル・カード・カレンダー・ギャラリー・タイムラインの5種類のビューで表示できる（OOUIパターンの完全実装）

### Figma

- **左パネル**（コレクションビュー）：レイヤー一覧、コンポーネント一覧
- **キャンバス**：選択可能なオブジェクトが直接操作可能
- **右パネル**（インスペクターパネル）：テキストを選択すると文字設定、コンポーネントを選択するとプロパティ変更オプションが表示されるコンテキストセンシティブUI

## OOUIと関連領域

### Domain-Driven Design（DDD）との対応

DDDとOOUIは「ドメインオブジェクトを中心に据える」という思想を共有している：

| DDD概念 | OOUI対応 |
|---------|---------|
| Entity（エンティティ） | UIオブジェクト |
| Aggregate（集約） | オブジェクトとその関連プロパティ |
| Bounded Context（境界付けられたコンテキスト） | ルートナビゲーションのドメイン分割 |
| Repository | コレクションビューのデータソース |
| Value Object | インスペクターパネルで編集される属性値 |

### RESTful APIとの対応

RESTfulとOOUIは「リソース（名詞）をアーキテクチャの中心に置く」という点で本質的に一致する：

| RESTful API | OOUI |
|------------|------|
| `GET /issues`（コレクション） | Issueコレクションビュー |
| `GET /issues/{id}`（シングル） | Issue詳細ビュー |
| `PATCH /issues/{id}`（更新） | インスペクターでのプロパティ変更 |
| `POST /issues/{id}/comments`（サブリソース） | シングルビュー内のコメントセクション |

この対応関係がフロントエンドとバックエンドの開発コミュニケーションを大幅に簡素化する。

### Alan Cooper「About Face」との関係

| About Faceのアプローチ | OOUIのアプローチ |
|----------------------|----------------|
| 「なぜそのUIが必要か（ユーザーの目標）」に答える | 「どのようにUIを構造化するか」に答える |
| ペルソナと目標（Goal）が設計の起点 | オブジェクトとビューの配置が設計の核心 |
| Goal-Directed Design で方向性を決める | OOUIで構造と操作フローを設計する |

### OOUX（Object-Oriented UX）/ Sophia Praterの拡張

UXデザイナーSophia Praterが2013年にA List Apart誌で発表した現代的OOUI手法：

1. **オブジェクト抽出**：プロジェクトブリーフから名詞を取り出し、繰り返し登場するものをオブジェクトとして特定
2. **コアコンテンツの定義**：各オブジェクトの要素を「コアコンテンツ」と「メタデータ」に分類
3. **オブジェクトのネスト**：オブジェクト間の関連をマッピングし、ナビゲーションを設計
4. **強制ランキング**：オブジェクト内要素を重要度順にランク付け

OOUXとOOUIの関係：OOUIは「画面とデータのマッピング（設計出力物）」であり、OOUXはその前段階の「情報アーキテクチャ・オブジェクトモデルの設計プロセス」。

## OOUIとCRUDインターフェースの違い

| 観点 | CRUD UI | OOUI |
|------|---------|------|
| **設計の起点** | データの4操作（作成・読取・更新・削除） | ドメインオブジェクトとそのビジネス的意味 |
| **インタラクション順序** | フォームを開いてからデータを操作 | オブジェクトを選んでからアクションを選ぶ |
| **ユーザーの意図** | UIから読み取れない（フィールドを変えた、としかわからない） | アクション名がビジネス的意図を表す（「キャンセル」「アーカイブ」） |
| **ナビゲーション** | CRUD操作のメニュー（作成、一覧、編集） | オブジェクト種別のコレクション→シングルへの遷移 |

**重要な注意点**：OOUIはCRUDと対立するものではない。OOUIのシングルビューではインプレースでの編集（Update）が自然に発生する。OOUIはCRUDをユーザーに対して「オブジェクトとその操作」として適切に包んで提示する設計思想である。

## OOUIのアンチパターン

### 1. 機能起点設計（最大の失敗）

**症状**：メニューに「〜する」という動詞形のアイテムが並ぶ。同じオブジェクトに関する操作が複数の別画面に分散する。

**処方**：まずオブジェクトを特定し、オブジェクト単位でUIを整理する。

### 2. 過剰なウィザード（モーダルな手順強制）

**症状**：「ステップ1/5」という進行表示。戻るボタンを押すと入力が消える。途中での変更・離脱が困難。

**原則違反**：GUIの基本思想は「オブジェクトをユーザーが自由に操れること」。線形のタスクフローへの閉じ込めはこれに反する。

**処方**：1つのオブジェクト詳細画面でインプレース編集を許可し、必要な情報を同一画面に集約する。

### 3. コンウェイの法則の罠（縦割りUI）

**症状**：タブが部門名になっている（「営業部」「開発部」「総務部」）。同一顧客の情報が複数サービスに分散。

**処方**：ユーザーが実際に扱うオブジェクト（「顧客」「案件」「契約」）を中心に設計し直す。

### 4. 隠れたアクション（ホバー依存）

**症状**：リストアイテムの操作ボタンがホバー時のみ表示。ユーザーがアクションの存在に気づかない。

**原則違反**：OOUIではオブジェクトが「自己表現」し、何ができるかが常に見えていなければならない。

### 5. 動詞ナビゲーション

**症状**：ルートナビゲーションのラベルが動詞形（「送信する」「管理する」「レポートを出力する」）。

**処方**：ナビゲーション項目は名詞形（オブジェクト種別）で表現する。

### 6. CRUDの素朴な露出

**症状**：「レコードを作成する」「レコードを更新する」「レコードを削除する」という機械的なフォームUI。ユーザーの業務的な意図が表現されない。

**処方**：ドメインに即したアクション名（「注文をキャンセルする」「スタッフをアーカイブする」）を使う。

## AIデザインプロセスへの関連性

- **オブジェクト抽出の自動化**：タスク記述文書やユーザーインタビュー記録から、AI が自動で名詞を抽出してオブジェクト候補を列挙するパイプライン
- **タスク指向UIの自動判定**：既存のUIスクリーンショットや設計仕様から、動詞起点のタスク指向UIを AI が検出・フラグアップするレビューシステム
- **DDDとOOUIの整合性チェック**：バックエンドのエンティティモデルとフロントエンドのUIオブジェクトが一致しているかを AI が自動検証
- **コレクションビュー/シングルビューの自動生成**：データモデルを入力として AI が OOUI的な画面構造を自動提案するコード生成ツール
- **ワークアウト採点**：本書の演習（タスク指向→OOUI変換ワークアウト）の解答を AI が自動評価・フィードバックするシステム
