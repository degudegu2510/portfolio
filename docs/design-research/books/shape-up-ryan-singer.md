## 概要

37signals（現 Basecamp）のプロダクト開発責任者 Ryan Singer が、同社が17年間かけて育てた**独自の製品開発プロセス**を体系化した書籍。アジャイル・スクラムとは異なるアプローチで、チームが意味のある仕事を確実に届けることを目的とする。

書籍は Basecamp のウェブサイトで**完全無料公開**されている。

## 3つのフェーズ

| フェーズ | 担当 | 説明 |
|----------|------|------|
| **Shaping（形作る）** | 小規模な上級チーム | 開発サイクル開始前に、問題とソリューションの輪郭を定義する。「どれだけ時間をかけるか」も決める |
| **Betting（賭ける）** | ステークホルダー | 形作られたプロジェクト候補の中から次のサイクルで取り組むものを選択。選ばれなかったものは消える |
| **Building（作る）** | 統合チーム | デザイナーとエンジニアが協力し、6週間のサイクルで自律的に完成させる |

## 主要な概念の詳細

### Appetite（アペタイト）

従来の見積もり（「どれだけかかるか？」）とは逆転した概念：

**「このアイデアに、どれだけの時間を投資したいか？」**

- 小さな仕事：2週間アペタイト
- 大きな仕事：6週間アペタイト

時間を固定し、スコープを柔軟にする（アジャイルのベロシティ重視とは異なる）。

### Fat-Marker Sketch（太マーカースケッチ）

ワイヤーフレームより粗い、**意図だけを伝えるスケッチ**。

- 太いペン（比喩）で描くことで細部の決定を先送りにし、余白を残す
- 「視覚的スタイリングなしで、非常に低い忠実度で描かれた UI コンセプト」
- チームが実装時に判断できる余地を意図的に残す

### Breadboard（ブレッドボード）

視覚的スタイリングなしに、**アフォーダンスとその接続を定義するテキストベースの設計図**。

```
[場所（Places）]: 画面・ページ・ダイアログ
[アフォーダンス（Affordances）]: ボタン・フィールド・リンク
[接続（Connection lines）]: 操作後の遷移
```

Figma のような高忠実度ワイヤーフレームではなく、接続関係だけを整理するツール。

### No Backlogs（バックログを持たない）

ベッティングテーブルで選ばれなかったアイデアは**廃棄する**。次のサイクルで必要なら再提案する。これにより：
- 古い・不要なアイデアの蓄積を防ぐ
- 常に今必要なものに集中できる

### クールダウン（Cooldown）

各6週間サイクルの後に設ける**2週間の自由期間**。バグ修正・技術的負債の解消・新しいアイデアの探索に使う。

## Shaping の5要素

| 要素 | 内容 |
|------|------|
| **Problem（問題）** | 解決すべき問題・チャンスを具体的に定義 |
| **Appetite（アペタイト）** | かける時間の上限（2週間 or 6週間） |
| **Solution（解決策）** | Fat-Marker Sketch や Breadboard で輪郭を描く |
| **Rabbit Holes（落とし穴）** | 事前に特定された難しい問題・潜在的な問題 |
| **No-gos（除外事項）** | 意図的にスコープから除外するもの |

## AIデザインプロセスへの関連性

- Shaping フェーズの「問題定義 → ソリューションの輪郭」プロセスを AI が支援（ユーザーデータからの問題抽出・Rabbit Holes の自動識別）
- Fat-Marker Sketch の生成を AI が担うワークフロー（Figma AI・Uizard・Galileo 等）
- Appetite の設定に過去のプロジェクトデータを AI が分析して参照できる
- Breadboard のテキスト設計は LLM との相性が高く、自然言語から自動生成できる
- バックログを持たない哲学は AI が提案する機能の即時評価・廃棄サイクルとも合致する
